# ДЗ1 — постановка задачи, baseline, протокол оценки, репозиторий

## О чем это ДЗ

Это домашнее задание про **нормальную R&D-постановку задачи в ML**:

- корректно сформулировать задачу;
- зафиксировать гипотезу;
- собрать **baseline**;
- продумать **протокол валидации** без утечек;
- оформить решение в виде **воспроизводимого репозитория**.

Главная цель: показать, что вы умеете **ставить ML-задачу как исследователь/инженер**.

---

## Формат

- Работа **индивидуальная**.
- Каждый студент выбирает **одну** задачу из списка ниже.
- **Повторения задач внутри группы запрещены.** Кто первым занял задачу — того и задача.
- Можно брать только задачи из списка ниже, если отдельно не согласовано иное.

---

## Что нужно сдать

Нужно сдать **GitHub-репозиторий** с решением.

В репозитории должны быть:

1. `README.md` с полным описанием задачи, baseline и протокола оценки.
2. Код, который **запускается**.
3. Инструкция по запуску.
4. Конфиг(и) эксперимента.
5. Скрипт(ы) обучения и оценки.
6. Зафиксированный результат baseline на локальной валидации.
7. Краткие выводы и интерпретация результата.

## Что должно быть в README

Ваш `README.md` должен содержать следующие разделы (не все пункты в них обязательные, но если что-то пропускаете - будьте готовы объяснить почему оно не нужно):

### 1. Постановка задачи
Нужно описать:
- что именно предсказывается;
- какой объект предсказания;
- какие данные доступны на момент предсказания;
- почему задача имеет практический смысл;
- какие ограничения у задачи;
- какая **основная гипотеза** проверяется.

Пример хорошей гипотезы:
> Если использовать TF-IDF + Logistic Regression вместо случайного baseline, то F1 на локальной валидации вырастет, потому что в задаче важны лексические паттерны.

Пример плохой гипотезы:
> Попробую разные модели и посмотрю, что лучше.

### 2. Формализация задачи
Нужно явно указать:
- `X` — что является входом;
- `y` — что является целевой переменной;
- тип задачи: binary / multiclass / multilabel / regression / ranking / forecasting;
- единицу наблюдения (sample / object / patient / user / document / image / time step);
- что считается одним предсказанием;
- какие признаки доступны на inference.

### 3. Данные
Нужно кратко описать:
- источник данных;
- размер данных;
- типы признаков;
- пропуски;
- дисбаланс классов (если есть);
- потенциальные источники шума в разметке;
- потенциальные источники leakage (лики).

### 4. Baseline
Нужно описать:
- какой baseline выбран;
- почему именно он;
- почему он является **честным** baseline, а не «почти SOTA»;
- какие preprocessing steps используются;
- какие гиперпараметры зафиксированы.

Baseline должен быть **достаточно простым**, чтобы:
- его можно было быстро запустить;
- его можно было интерпретировать;
- с ним можно было честно сравнивать будущие улучшения.
- я, т.е преподаватель, не устал его проверять (как и вы при следующих иттерациях)

### 5. Протокол оценки
Нужно описать:
- основную метрику;
- почему именно она;
- как формируется train/val split;
- почему такой split корректен;
- какие есть риски ликов;
- какие дополнительные метрики полезно смотреть;
- как выбирается лучшая модель.

Не все пукнты тут обязательны, но если что-то пропускаете - будьте готовы объяснить почему.

### 6. Результаты baseline
Нужно показать:
- итоговую метрику на локальной валидации;
- при наличии — среднее и стандартное отклонение по нескольким запускам / фолдам;
- краткую таблицу результатов;
- 2-3 примера типичных ошибок модели.
- релевантные графики и краткое описание

### 7. Выводы
Нужно коротко ответить:
- что получилось;
- где baseline слаб;
- какая следующая гипотеза выглядит наиболее разумной;
- какие ограничения у текущего решения.

## Обязательные вопросы, на которые нужно дать ответ

Ниже список вопросов, на которые в README должен быть дан **явный ответ**.

1. **Что именно нужно предсказывать и почему это такая постановка задачи?**
2. **Что является объектом наблюдения, что входит в `X`, что является `y`, и какие данные доступны на inference?**
3. **Какая метрика и какой validation protocol выбраны, и почему они корректны для этой задачи?**
4. **Какой baseline выбран, какие preprocessing steps и признаки он использует, и почему это разумная точка отсчета?**
5. **Какие основные ошибки делает baseline, есть ли риск лика, и какая следующая гипотеза выглядит наиболее разумной?**

## Это R&D-курс, поэтому вы должны не только отвечать на вопросы, но и **ставить их сами**.

Нужно сформулировать **минимум 3 собственных исследовательских вопроа** по вашей задаче.

Эти вопросы должны быть не общими, а проверяемыми. Они должны вести к экспериментам или к изменению протокола оценки.

### Примеры хороших исследовательских вопросов
- Улучшает ли стратифицированный split стабильность оценки по сравнению с одним holdout?
- Дает ли TF-IDF + linear model более устойчивый baseline, чем маленький transformer при ограниченном бюджете?
- Влияет ли балансировка классов на recall редкого класса?
- Улучшают ли аугментации качество на сложных примерах или только подгоняют train?
- Насколько чувствителен результат к seed?
- Есть ли систематические ошибки на определенных подгруппах данных?
- Меняется ли качество, если убрать потенциально шумные примеры?
- Помогает ли лог-преобразование таргета в регрессии?
- Что происходит с качеством при переходе от random split к group/time split?
- Насколько сильна разница между «очень простым» baseline и «чуть более сильным, но все еще честным» baseline?

### Примеры плохих вопросов
- Какая модель лучше?
- Можно ли улучшить качество?
- Что будет, если попробовать нейросеть?

Плохие вопросы плохи тем, что они:
- неоперационализированы;
- не задают критерий сравнения;
- не фиксируют, что именно проверяется.

## Требования к baseline

Baseline должен быть **простым, корректным и воспроизводимым**.

### Хороший baseline
- **Содержит EDA-блок (хотя бы короткий)**
- **Табличные данные:** Logistic/Linear Regression, Ridge/Lasso, CatBoost/LightGBM с простыми настройками.
- **Тексты:** TF-IDF + Logistic Regression / Linear SVM / Ridge; простой transformer baseline, если он честно воспроизводим.
- **Изображения:** pretrained ResNet/EfficientNet с простой head, небольшой CNN.
- **Временные ряды:** seasonal naive, moving average, lag features + простая модель.
- **Аудио:** log-mel spectrogram + простой классификатор, либо pretrained audio backbone + linear head.

### Плохой baseline
- слишком сложное heavy solution или ансамбль без объяснения;
- решение, которое трудно запустить;
- Kaggle notebook без воспроизводимого пайплайна;
- baseline без обоснования, почему это точка отсчета.

## Требования к протоколу оценки

Протокол оценки — одна из главных частей ДЗ.

### Обязательно
- основная метрика должна соответствовать задаче;
- нужна **локальная валидация**, а не только public leaderboard;
- split должен быть корректным: stratified / group / time-based — в зависимости от природы данных;
- preprocessing без лика: всё обучается **только на train**;
- выбор лучшей модели — **только по validation**.

### Для сильной работы желательно
- k-fold / repeated validation;
- `mean ± std` по фолдам или сидам;
- дополнительные метрики;
- error analysis по срезам.
- бутсрап и оценка доверительного интервала

### Ошибки
- случайный split для time series;
- preprocessing на всех данных до split;
- подбор модели по test/public LB;
- сравнение моделей на разных split;
- отсутствие описания расчета метрики.
